Scalabilty in Distributed Ledger Technology

Thanks to John Adler and Ryan Berryhill who explained this to me.

Everybody knows it... cryptocurrencies have problems with scalability. To fix scalability, we need to replace the slow, power-hungry Proof-of-Work with the fast, svelte Proof-of-Stake, right?

WRONG! Proof-of-Work is not the source of the scalability problem, and Proof-of-Stake will not solve it!

Here is how I define the scalability problem:

The amount of work each node in a distributed ledger must do is O(n) in the number of transactions on the network.

The strong safety guarantees in the face of total decentralization carry a cost: each and every node must validate each and every transaction. Defined in this way, we can distinguish two types of scalability solutions:

  • Not forcing all nodes to validate all transactions
  • Somehow changing the O(n) time to validate all transactions, maybe through parallelization.
  • (Secret option: combine both of these)

Not forcing all nodes to validate all transactions: Sharding

Sharding makes use of the (fairly obvious) idea that maybe we really don't need every single node to validate every single transaction. One thousand valdidations ought to be enough for anyone! Maybe we can "shard" the network into smaller pieces that each take care of their own subset of all transactions?

Implementing sharding is really not easy, mind you. How do you select which nodes will validate a particular transaction? What happens if one ore more of those nodes go offline or are dishonest? How will transaction fees be distributed?

Ethereum's end goal has always been to incorporate sharding, and they have a great big FAQ page which should satisfy your immediate curiosity.

Improving O(n) Validation Time

Consider this: if validating transactions was instantaneous, then maybe blockchain wouldn't suck. In this case, blocks could be arbitrarily large, and get validated in seconds. You would have all the benefits of a global, totally decentralized ledger, without the comparatively high costs and slow transaction times.

We can further classify this class of solutions into two categories:

  • Reducing validation work by a constant factor
  • Reducing the complexity class

Reducing Work by a Constant Factor

The most immediately obvious method of reducing validation work is to parallelize it. Vitalik has suggested doing this for Ethereum, but Ethereum really does not lend itself well to parallelization.

One idea would be to create an all-new blockchain platform specifically designed to accommodate a high degree of parallelizability. If a smart contract language is desired, this would require designing an all-new VM, possibly disallowing Turing-completeness.

A complementary idea would be some kind of hardware acceleration.

Reducing the Complexity Class

This seems unlikely, and I certainly have no ideas for how to do this. Basically, if you could somehow validate a list of n transactions in O(lg(n)) time or something, then you would have solved the scalability problem very directly.

Secret Option: Both

To be sure, either one of the previously presented solutions would probably be sufficient. However, it's interesting to note that developments in either area are orthogonal; that is, they could easily be used at the same time, offering even faster/cheaper transaction validation.
